# meeting-minutes

將使用者提供的一或多份會議逐字稿（口語化、高雜訊原始文字）整理成結構化 Markdown 會議紀錄，覆寫寫入 `Data_Analysis/weekly_report/inputs/meeting_notes.md`，供週報系統（`generate_report.py`）讀取。

## 呼叫方式

使用者可提供：
- 一或多個逐字稿 `.md` 檔的路徑（可一次給多個）
- 直接在對話中貼上逐字稿文字
- 兩者混合

若使用者未提供任何逐字稿，詢問他們要提供路徑還是直接貼上文字。

## 執行流程

1. **收集輸入**：確認所有逐字稿來源（路徑或文字）。若為路徑，用 Read 工具逐一讀取。
2. **轉換每場會議**：對每份逐字稿套用下方【轉換規則】，產生結構化紀錄。
3. **合併輸出**：將所有會議紀錄依序排列，多場之間以 `---` 分隔，**完整覆寫**寫入 `Data_Analysis/weekly_report/inputs/meeting_notes.md`。
4. **確認完成**：告知使用者已整理幾場會議，並列出每場的名稱與日期。

## 轉換規則

閱讀每份逐字稿時，嚴格遵守以下原則：

1. **嚴格客觀**：只記錄逐字稿中實際發生的事實與結論，嚴禁加入主觀評論或推論。
2. **廢話過濾**：自動忽略問候、閒聊、結巴、重複、口頭禪或無意義的討論。
3. **任務綁定**：任何待辦事項（Action Item）必須明確標註「負責人」與「截止日期/時間點」。若逐字稿中未明確提及，負責人寫「[待確認/請指派]」，截止日寫「[待確認]」。
4. **專業術語**：保留正確的技術專有名詞、產品名稱與專案術語，切勿隨意翻譯或簡化。

## 每場會議的輸出格式

```markdown
# [會議名稱] - 會議紀錄

* **Date**: [根據逐字稿判斷，若無則填今天，格式 YYYY-M-D]
* **Hosts**: @[主持人英文ID] [中文姓名]
* **Attendees**: @[與會者1英文ID] [中文姓名]　@[與會者2英文ID] [中文姓名]
* **Recording**: [錄影連結，若無則留空]

## 討論摘要 (Minutes)
- [議題 A 簡述，一句話]
- [議題 B 簡述，一句話]
- [議題 C 簡述，一句話]

## 關鍵決策 (Decisions)
1. [關鍵決策 A]
2. [關鍵決策 B]

## 待辦事項 (Next Action Item)
- ☐ [待處理任務] => @[英文ID] [中文姓名]（截止：YYYY-MM-DD）
- ☑ [已完成任務] => @[英文ID] [中文姓名]

## 附件 (Attachments)
- [文件名稱]：[連結或路徑]
```

### 格式說明

- **討論摘要**：每個主要議題一條，一句話精煉，不加細節。
- **關鍵決策**：僅記錄明確拍板的決策（有具體結論或共識），不記錄討論過程。
- **待辦事項**：`☐` 表示待處理，`☑` 表示已完成；每條必須有 `=>` 後接負責人，截止日若有明確提及則加在描述末尾（格式：`（截止：YYYY-MM-DD）`）。
- **附件**：若逐字稿中無提及任何文件/連結，此 section 省略不輸出。

## 輸出規格

- **寫入路徑**：`Data_Analysis/weekly_report/inputs/meeting_notes.md`（每週覆寫，不累積歷史）
- **多場會議分隔**：每場之間插入 `---`
- **語言**：繁體中文（技術術語保留英文）
- **不保留原始 HTML 註解**：輸出檔只包含整理好的會議紀錄，不保留原本的使用說明 block
